## 幻灯片 1 — JUNO DCI 监控升级 · 0:30

大家下午好。我是韩骁，代表 DCI 小组进行汇报。今天我介绍 JUNO 分布式计算监控方面的近期工作。报告包括 Git 中的仪表盘配置、集中式组件日志、通过 MCP 访问 Grafana，以及 AI Agent 在仪表盘工作流程中的使用。

## 幻灯片 2 — 概览 · 0:55

先看一下整体内容。监控系统不只是显示图表。它还服务于 JUNO 数据重处理，帮助我们发出告警、定位故障和恢复服务。这次有三项技术更新。仪表盘 JSON 保存在 Git 中。组件日志集中保存在一个系统中。Agent 可以通过 MCP 访问 Grafana。最后，我会说明 AI 如何支持完整的仪表盘工作流程。

## 幻灯片 3 — 让仪表盘可以复现 · 0:10

我先介绍仪表盘配置。目标是把仪表盘定义保存在 Git 中，并用统一的方式加载到 Grafana。

## 幻灯片 4 — 仪表盘保存在 Git 仓库中 · 1:00

左边是以前的方法。我们在 Grafana 页面中编辑仪表盘。Grafana 把结果保存在自己的数据库中。需要时，我们再进行手动备份。这种方法可以用于恢复，但每次修改没有单独的审查过程。右边是现在的方法。我们仍然可以在 Grafana 中编辑，也可以直接修改 JSON。之后，我们在 Git 中审查 JSON 差异。Provisioning provider 再把审查后的文件加载到 Grafana。现在，Git 中保存的是仪表盘定义。

## 幻灯片 5 — 仪表盘文件和交付流程 · 1:05

仓库中现在有 31 个仪表盘文件。它们分在五个 provider 下：Admin、DIRAC、TPC、User 和 Shift。整个流程有四步。我们创建仪表盘，把它保存为 JSON，在 Git 中审查，然后由 Grafana 加载。Grafana 每 30 秒检查一次 provider。我们仍然允许在 Grafana 页面中编辑。所以，页面中的修改必须回到 Git。我们需要导出、审查并提交。否则，Grafana 和仓库中可能是不同的版本。

## 幻灯片 6 — 配置方法 · 0:50

这一页展示主要的配置步骤。第一，注册一个基于文件的 provisioning provider。它指向 TPC 仪表盘目录，每 30 秒检查一次更新。第二，把 provisioning 目录挂载到 Grafana 容器。第三，从 grafana.db 中导出现有仪表盘，并保存为 JSON 文件。完成这些步骤后，Grafana 就可以从仓库加载仪表盘定义。

## 幻灯片 7 — 把组件日志保存到数据库 · 0:10

第二部分是组件日志。指标显示服务发生了变化。日志帮助我们找到与这个变化相关的组件事件。

## 幻灯片 8 — 集中式组件日志 · 1:05

JUNO DIRAC 运行了 60 多个组件。默认情况下，每个组件把日志写到本地磁盘。在 CAS_Prod.cfg 中，我们为 services 和 agents 保留 stdout，同时增加 mqLogs。mqLogs backend 把日志消息发送到 ActiveMQ。Logstash 读取并解析消息，然后写入 Elasticsearch。Grafana 查询这个集中式存储。这样，services 和 agents 使用相同的日志流程，同时仍然保留本地输出。

这是实时的 Component Logs 仪表盘。它的加载速度可能比较慢，所以我先等一下。左边列出了三个排查步骤。右边是实时仪表盘，可以在这个区域中上下滚动。

[演示：等待页面加载，然后滚动查看三个面板。]

第一个面板显示 information、warning 和 error 消息的数量。第二个面板显示这些数量随时间的变化。第三个面板显示单条日志记录。我们先检查消息级别，再选择时间范围，最后查看相关记录。Category、Name 和 Level 三个筛选条件会同时应用到所有面板。

## 幻灯片 10 — 为 Agent 提供受控访问 · 0:10

第三部分是 MCP 访问。Agent 可以使用监控数据，但请求必须经过身份认证和权限检查。

## 幻灯片 11 — 通过 MCP 访问 Grafana · 1:10

客户端不直接连接 Grafana。它把 MCP 请求发送到 mcp.ihep.ac.cn。这里是统一网关。网关检查 API key 和对应的权限范围。认证服务使用 IHEP LDAP 和 PostgreSQL。如果请求通过检查，网关才把工具调用发送给 mcp-grafana。这个服务使用 Grafana API 和渲染接口。Grafana 从 Prometheus 和 Elasticsearch 读取数据。在这个设计中，身份和权限策略由统一网关管理。Grafana 相关操作由 mcp-grafana 服务处理。

## 幻灯片 12 — 从运维问题到证据 · 1:00

这里给出一个例子。用户问，为什么某一对站点之间的 TPC push 传输失败。首先，网关检查用户身份和请求范围。然后，Agent 可以用 search_dashboards 和 get_dashboard_summary 找到仪表盘。它可以用 get_dashboard_panel_queries 读取面板查询，并在需要时用 query_prometheus 查询数据。它还可以用 get_panel_image 返回面板图片，或者用 generate_deeplink 返回 Grafana 链接。这些名称来自当前 IHEP MCP 的 tools/list。它们用于检查和解释。写操作仍然需要明确授权。

## 幻灯片 13 — AI 工作流程 · 0:10

最后一部分介绍 AI Agent 如何支持从仪表盘创建到结果检查的整个流程。

## 幻灯片 14 — AI 支持监控工作流程 · 1:00

运维人员先定义仪表盘的含义和阈值。Agent 收到请求后创建仪表盘 JSON。我们构建结果，并审查 Git 差异。审查后的文件再通过 provisioning 加载到 Grafana。Agent 可以通过 MCP 服务检查结果。如果发现问题，结果会回到 Agent，再进行一次修改。Agent 负责重复工作。运维人员仍然负责定义含义、批准结果，并决定执行什么操作。

## 幻灯片 15 — TPC 传输矩阵 · 1:15

这是实时的 TPC 传输矩阵。它基于 Albert Dzakhoev 的工作。我们使用 AI 改进了矩阵排序，并增加了平均成功率面板。仪表盘包括 pull、push、streamed 和 combined 四种传输模式。

[演示：上下滚动矩阵，并比较两种传输模式。]

一次失败的传输只是一个事件。如果同一行或同一列重复出现失败，问题可能和某个站点有关。如果只有一种模式出现相同问题，问题可能和传输方法有关。平均成功率可以帮助我们判断问题是短时间出现，还是持续存在。这个矩阵汇总了多次传输测试的结果。

## 幻灯片 16 — 已完成工作和下一步 · 1:00

左边列出了目前已经完成的工作。我们有 31 个通过 provisioning 管理的仪表盘文件、TPC 传输视图、Grafana 的 MCP 访问方式，以及集中式组件日志。右边列出了下一步工作。告警需要负责人和处理链接。仪表盘需要构建、schema 和截图检查。MCP 操作应从只读权限开始，并经过明确授权。我们还希望从指标跳转到日志时，保留站点、组件和时间范围。

## 幻灯片 17 — 总结 · 0:50

最后总结三点。第一，Git 和 provisioning 让我们可以审查和恢复仪表盘定义。第二，TPC 视图和组件日志为问题诊断提供数据。第三，运维人员和 Agent 可以通过不同方式使用同一份 Grafana 数据。运维人员直接使用 Grafana。Agent 通过 IHEP MCP 网关访问。主要更新是从监控信号到运维操作的整个流程。
