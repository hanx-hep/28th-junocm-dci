# 从仪表盘到诊断

## 15 分钟英文演讲脚本

时间均为大约估算，包含两次现场 Grafana 视图展示的短暂停顿。总计划时间约为 15 分钟。

## 幻灯片 1 — 从仪表盘到诊断 · 0:35

大家下午好。我是韩骁，代表 DCI 小组进行汇报。今天我将介绍 JUNO 分布式计算基础设施的最新监控升级。标题"从仪表盘到诊断"概括了这项工作的方向。我们不仅是在增加更多的图表，更是在连接配置、监控证据和受控的 Agent 访问，使运维症状能够更快速地导向有用的诊断和清晰的下一步行动。

## 幻灯片 2 — 从信号到行动的更短路径 · 0:55

本页幻灯片展示了核心信息。监控正在成为一个运维系统，而非独立仪表盘的集合。我们正在改进三个特性。第一，可复现：仪表盘定义和配置均保存在 Git 中。第二，可诊断：指标展示症状，而集中式日志解释背后的事件。第三，可访问：运维人员直接使用 Grafana，Agent 可以通过受控的 IHEP MCP 网关访问相同的证据。目标很简单——缩短从发现问题到知道下一步有用行动的时间。

## 幻灯片 3 — 一个监控闭环 · 1:10

这张图展示了完整的架构。左侧是运维信号：服务和主机指标、TPC 传输记录、DIRAC 组件日志。指标进入 Prometheus。传输记录和集中式日志可通过 Elasticsearch 获取，组件日志经过 ActiveMQ 和 Logstash。Grafana 将这些数据源整合为运维视图。Git 中的仪表盘 JSON 被配置到 Grafana 中，因此可视化层也是受控配置。运维人员直接查看 Grafana。AI Agent 通过集中式 MCP 网关和 mcp-grafana 适配器访问 Grafana。这就是我将讨论的三项升级：可复现的基础设施、更丰富的证据和受控的接口。

## 幻灯片 4 — 让仪表盘可复现 · 0:15

我将从基础开始：让仪表盘可复现。这听起来像是一个配置细节，但它改变了我们审查、部署和恢复监控环境的安全程度。

## 幻灯片 5 — Provisioning 移动了控制点 · 1:05

以前，正在运行的 Grafana 实例实际上是主要的事实来源。我们在 UI 中编辑仪表盘，Grafana 存储状态，手动备份在需要恢复时提供保护。备份是有用的，但它没有为每次变更提供清晰的审查路径。在当前工作流中，仪表盘 JSON 存储在 Git 中，作为普通 diff 进行审查，并通过 provisioning provider 加载。这使得监控布局可重建，且变更在部署前可见。底部的区分很重要：备份保护过去，而 provisioning 控制下一次变更。

## 幻灯片 6 — 仓库作为控制面 · 1:10

该仓库目前包含 31 个仪表盘文件，分布在五个提供商下：Admin、DIRAC、TPC、User 和 Shift。交付循环是创建、捕获、控制和同步。运维人员或 Agent 可以组合仪表盘，但结果必须以仪表盘 JSON 的形式返回。Git 是审查边界，Grafana 每 30 秒刷新一次 provider。有一个注意事项需要关注：为方便起见，UI 更新仍然被允许。因此，在 Grafana 中进行的编辑必须导出、审查并提交。否则，运行实例可能会偏离仓库。Git 仍然是持久的事实来源。

## 幻灯片 7 — 实时 TPC 传输矩阵 · 1:20

这是实时的 TPC 传输监控仪表盘。加载可能需要一点时间。顶部可以按时间间隔、源站点、目标站点、成功状态和复制模式进行筛选。该仪表盘包含 12 个面板：8 个表格和 4 个状态时间线，涵盖拉取、推送、流式和组合模式。

[演示：垂直滚动浏览矩阵，并比较至少两种复制模式。]

矩阵的价值在于关联性。单个失败的传输只是一个事件。重复的行、列或模式模式表明问题跟随某个站点、方向或传输方法。平均值面板也有助于区分瞬时故障和持续性能下降。这将许多单独的测试记录转化为紧凑的运维信号。

## 幻灯片 8 — AI 在审查循环中 · 0:55

TPC 仪表盘也展示了 AI 辅助的用武之地。运维人员定义语义、传输模式、阈值以及颜色的含义。然后 Agent 可以处理重复性的仪表盘组合工作：面板结构、查询、转换、变量和布局。输出在配置之前被构建、diff 和审查。最后，运维人员验证运维视图。因此，重要的结果不是 AI 制作了一个仪表盘，而是在保持正常的 Git 审查边界和明确的人工控制的同时，实现了更快的组合速度。

## 幻灯片 9 — 受控的 Agent 访问 · 0:15

第二项升级是受控的机器访问。一旦监控证据结构化，Agent 就可以使用它——但必须通过清晰的身份验证和授权边界。

## 幻灯片 10 — MCP-Grafana 架构 · 1:15

客户端不直接连接 Grafana。它向集中式网关 mcp.ihep.ac.cn 发送 MCP JSON-RPC 请求。网关通过身份验证服务验证密钥和范围，该服务连接到 IHEP LDAP 和 PostgreSQL。只有经过授权的工具调用才会被转发到 mcp-grafana。在监控边界内部，mcp-grafana 使用 Grafana API 和渲染接口。Grafana 读取 Prometheus 和 Elasticsearch，并可以使用图像渲染器获取可视化上下文。这种分离带来了两个有用的特性：身份和策略保持集中，而 Grafana 特定的行为保持在专用适配器内部。

## 幻灯片 11 — 从问题到证据 · 1:00

这里是一个具体的诊断路径。用户询问为什么某个站点对的 TPC 推送传输失败。网关首先验证调用者和范围。然后 mcp-grafana 查询相关数据，并可以渲染运维人员通常会查看的面板。Agent 返回证据并提出下一步诊断步骤。证据有两种形式：结构化的定义和查询结果，以及来自渲染面板的可视化模式。边界同样重要。Agent 收集和解释证据；修复仍然是明确的运维操作。MCP 是一条访问路径，而不是一个新的监控数据源。

## 幻灯片 12 — 将日志置于指标旁 · 0:15

第三项升级是将集中式组件日志置于指标旁。指标擅长显示行为发生了变化。日志则用于解释哪个组件事件导致了该变化。

## 幻灯片 13 — 集中式日志弥补了差距 · 1:00

DIRAC 组件将日志消息发布到 ActiveMQ。Logstash 消费并解析它们，在 Elasticsearch 中建立索引，Grafana 提供查询和可视化层。该流水线支持三个阶段的排查。指标回答"什么变了？"。时间线回答"何时开始的？"。单条记录回答"哪个组件解释它？"。统一的时间戳和通用过滤器定义了一个调查窗口，取代了逐一访问服务主机的需要。Prometheus 和 Elasticsearch 回答不同的问题，但 Grafana 将两者置于同一个运维工作流中。

## 幻灯片 14 — 实时组件日志视图 · 1:15

这是实时的组件日志仪表盘。它加载比 TPC 仪表盘慢，所以我给它一点时间。右侧的垂直面板已缩小，我们可以滚动查看。

[演示：当三个面板加载完成后，滚动浏览它们。]

第一个面板显示信息、警告和错误消息的分布。第二个显示数量随时间的变化。第三个展示单条记录。实际的调查从宽泛开始，然后逐步缩小：首先识别异常的错误组合，然后选择时间窗口，最后检查负责的消息。类别、名称和级别过滤器在所有三个视图中保持相同的上下文。

## 幻灯片 15 — 连接的运维闭环 · 1:00

这些升级现在形成了一个连接的闭环。TPC 或服务症状出现在 Grafana 中。我们查询相关指标和日志，建立基于证据的诊断，并将运维操作留给运维人员。MCP Agent 可以缩短证据收集和解释的时间，但它不会取代人的判断。事件结束后，经验教训可以以更好的仪表盘或告警定义的形式返回系统。由于该定义是代码，它经过审查和配置后才会成为下一个运维视图的一部分。

## 幻灯片 16 — 已完成与下一步 · 1:00

左侧总结了目前已交付的内容：包含 31 个 JSON 文件的仪表盘配置、12 面板 TPC 传输视图、通往 Grafana 的受控 MCP 路径，以及带有三个互补视图的集中式组件日志。右侧展示了下一步的增量。我们需要带有明确所有权的可操作告警、自动化的仪表盘发布检查、具有显式授权的只读 MCP 操作，以及携带站点、组件和时间上下文的从指标到日志的下钻链接。这些步骤将当前的架构转化为更可复现的运维实践。

## 幻灯片 17 — 三个要点 · 0:50

有三个要点。第一，监控环境更具可复现性，因为仪表盘通过 Git 进行审查和配置。第二，更具可诊断性，因为 TPC 视图暴露了模式，集中式日志提供了组件级别的上下文。第三，更具可访问性，因为相同的 Grafana 证据既服务于直接查看的运维人员，也通过 IHEP MCP 网关服务于 Agent。关键的升级不仅仅是又一个仪表盘，而是一条从信号到行动的更短且受控的路径。
